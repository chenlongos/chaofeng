## 十一、SimCar 模拟器全链路可行性补充评估

本轮结合 `deep-research-report.md`、本日志、`HANDOFF-2026-07-17.md`、当前 worktree 代码和 SimCar 在线 API 文档，对“绕过这个障碍物去捡球”进行了端到端对表。

### 1. 结论分层

SimCar 可以作为 AgentOS 第一阶段的功能级数字仿真，跑通以下真值驱动闭环：

```text
中文文本/语音
→ 意图路由
→ 复合任务编排
→ 底盘离散动作和抓放指令
→ 10 Hz 仿真状态反馈
→ 状态 Judge
→ 成功、重试或失败终态
```

但只使用当前公开接口，尚不能完整验证：

- “这个障碍物”的实例 grounding；
- 基于图像或地图的几何避障；
- RewardClassifier 真实视觉判定；
- VLA/WAM 基于相机观测输出连续动作；
- Nav2、MoveIt 2 或 ROS Action 级闭环。

当前接口已经提供 `up/down/left/right/stop/grab/release`、距离和角度参数、机器人 `x/z/rotation`、`hasBall`、`grabAvailable`、`isColliding`、`ballPosition`、reset，以及按 `clientId` 隔离的 REST/WebSocket 通信。这足以验证 AgentOS 的任务编排、命令执行、反馈判定和恢复逻辑。

当前主要缺口是：障碍物和目标区的结构化几何、相机帧导出、确定性场景 seed、命令完成 ACK/幂等、场景生命周期管理，以及连续控制或 ROS 接口。因此，当前 SimCar 应定位为“端侧控制闭环验证器”，不能把真值状态闭环表述为已经完成真实视觉或 VLA 能力。

### 2. 与深度研究报告的关系

深度研究报告提出的 `语言解析 + BT/PlanSys2 + Nav2 + 地图/感知 + MoveIt 2 + 抓取 + 闭环反馈` 仍适合作为最终目标架构，但不应在 SimCar 第一阶段全部照搬。

SimCar 当前是离散命令和真值状态接口，不是 ROS 机器人仿真器。端侧首版优先用轻量几何规划和状态 Judge 验证系统闭环；只有进入真实视觉、ROS 导航和机械臂轨迹验证时，才扩展 SimCar 或迁移到 Gazebo/Isaac Sim 等更合适的平台。

## 十二、模拟器侧与端侧责任边界

### 1. 模拟器侧由其他同事负责

建议由模拟器同事逐步补充：

- 稳定的仿真实例创建、销毁和 `clientId` 生命周期；
- 障碍物、球、桶、目标区的对象 ID、位姿和几何查询；
- 带 seed 的确定性 reset 和标准场景集；
- 状态时间戳、序列号、命令 ID、完成反馈和 cancel；
- RGB/深度图像流、相机内外参；
- 碰撞、抓取成功、球进入目标区等任务事件；
- 后续需要时再增加连续控制或 ROS bridge。

这些属于模拟器能力建设，端侧不在当前仓库内代替实现。

### 2. 端侧由我们负责，但在当前批次收口后启动

端侧后续探索分支建议包含：

1. `SimCarAdapter/Robot Gateway`：管理 `clientId`、Room、REST/WebSocket、单位换算、状态缓存、超时、停止和断线重连；
2. 三 ID 与审计：贯通 `session_id / task_id / request_id`，SimCar `clientId` 作为机器人实例 ID 单独管理；
3. 强类型命令和状态契约：兼容新增字段，不破坏既有 schema；
4. 复合任务执行器：实现“定位球→规划接近→导航→等待可抓→抓取→验证”；
5. 轻量 Route Planner：先消费真值几何，输出转角和前进距离，不在首版强行接 Nav2；
6. World Model 真值适配器：明确标记 `source=sim_ground_truth`，维护时间戳、实体状态和 `last_seen`；
7. `SimStateJudge`：根据 `hasBall`、`armState`、`isColliding` 等状态判定，和真实 RewardClassifier provider 严格分开；
8. 恢复策略：碰撞立即 stop、退后重规划、抓取失败重新接近、状态陈旧或断线安全失败；
9. 自动化测试：Room 隔离、命令转换、状态超时、reset、路径执行、抓取成功和失败恢复。

当前 `robot.pick_ball` 被定义为不触发路线规划的单步任务，`NavClient` 是恒成功桩，真实 RewardClassifier 仍是占位，VLA provider 面向 `lerobot-rollout` 真机命令。因此仿真接入应走新的 Robot Gateway/SimCar adapter 和复合任务路径，不应把 SimCar 协议直接混入当前真机 VLA provider。

## 十三、Git 收口与新分支决策

本轮决定：**先把当前 worktree 优雅收口并合入统一基线，再从干净基线创建 SimCar 端侧探索分支。不要直接从当前脏 worktree 横向开发仿真功能。**

推荐执行顺序：

1. 修正 `test_orchestrator_qa_prompt.py` 两处 `COMPLETE` 为既有契约 `complete`；
2. 清除 `__pycache__`，运行相关测试和全量测试，记录新的权威数字；
3. 区分真实业务改动与 CRLF/LF 行尾噪音，排除 `vla_service/**` 等无关变更；
4. 按“否定安全拦截、LLM fallback、Ollama provider、QA prompt/demo”等主题显式暂存和提交，禁止 `git add -A`；
5. 保证 `worktree-tests-scaffold` 工作树干净、提交历史可独立审阅；
6. 从正式团队目标分支创建临时 integration 分支，在该分支合并 `worktree-tests-scaffold`，解决冲突并重跑全量测试；
7. integration 分支验证通过后，再更新正式团队分支；未经明确授权不 push origin；
8. 从合并后的唯一权威提交创建 `feat/simcar-agent-loop`（名称可在建分支时确认）；
9. SimCar 分支只放端侧 Gateway、任务执行、World Model 适配、状态 Judge 和相关测试，不夹带旧 LLM 收口内容。

这种方式优于从当前 worktree 直接开探索分支，原因是：新分支拥有单一可追溯基线；仿真故障不会和 LLM、行尾噪音混在一起；后续可独立评审、回退或合并；也能避免主工作区、tests-scaffold 和仿真分支形成第三套漂移实现。

截至本补充记录，尚未修改功能代码、运行新测试、执行合并、创建分支或 push。下一步仍是完成当前 worktree 的 P0 收口。

## 十四、端侧大模型驱动与已知几何绕障 Goal 执行记录

在 `feat/simcar-agent-loop` 独立 worktree 上继续完成端侧探索，未修改模拟器、`vla_service/**`、脏主工作区，也未 push。目标不是让大模型直接猜底盘命令，而是让端侧 Qwen 生成受约束高层计划，再由确定性规划器和反馈控制器执行。

### 1. 本轮落地模块

- `agent_api.task_planner`：端侧 LLM 输出白名单 JSON 任务计划，拒绝多余文本、未知技能和越界参数；
- `TaskPlan/TaskStep`：向后兼容新增的结构化任务契约；
- `SafeMotionSkill`：前进、后退、左右转、停车，限制总距离、总角度和单段长度；
- `KnownScene`：只解析配置或后续可验证感知提供的障碍物，禁止模型编造坐标；
- `GeometricAvoidancePlanner`：按 30 cm 方形车的半对角线计算旋转扫掠半径，膨胀障碍物并比较左右绕行候选；
- `WaypointNavigator`：把路径点编译为转向和不超过 5 cm 的平移片段，每段后读取状态；
- `SimCarTaskPlanSkill`：执行 `move/turn/stop/navigate_around/pick_ball`，支持“绕过已知障碍物后捡球”；
- `SimStateJudge`：区分纯运动和捡球复合任务终态；
- `demo_simcar_agent.sh`：统一文本、SenseVoice 音频、Ollama/Qwen、Gateway 和 Agent 演示入口。

端侧大模型计划示例：

```json
{
  "goal": "绕过可乐瓶捡球",
  "steps": [
    {"skill": "navigate_around", "arguments": {"target": "可乐瓶", "side": "auto"}},
    {"skill": "pick_ball", "arguments": {}}
  ]
}
```

### 2. 安全边界

- `stop` 走确定性规则，不依赖 LLM 规划成功；
- 否定指令在 Planner 前拦截；
- Gateway 在动作前再次校验完整计划并解析全部目标；
- 未知目标、坐标缺失、距离过近、参数越界、碰撞、断连和超时均停车失败；
- 障碍物距车仅约 5 cm 时返回 `unsafe_initial_clearance`，不自动转向或盲目后退；
- Demo 固定 `max_retries: 1`，不进行碰撞后的盲目恢复；
- 当前障碍物来源明确标记为 `known_scene_geometry`，不能表述为视觉识别。

### 4. 当前诚实结论

现在可以完成“端侧大模型把自然语言转换成结构化复合计划，并在已知障碍物坐标下闭环绕障捡球”的软件 MVP。原有无障碍真值捡球已经在线验证；本轮新增的已知几何绕障尚未在一个经过测量的在线障碍场景中做 live 验收，因此不能宣称已完成真实视觉绕障。

要实现真正的“识别前面那个可乐瓶并绕过去捡球”，仍需 RGB/RGB-D 或模拟器对象接口、相机标定、检测与跟踪、坐标转换、观测时间戳和置信度。感知结果可直接接入 `KnownScene` 等价接口，现有高层 Planner、几何规划、反馈执行和 Judge 可以继续复用。
