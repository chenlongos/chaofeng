## 一、项目简介

**嘲风** 是一个面向**多部署环境**（桌面、实验室、温室、农田等）的语音控制机器人系统，形态为**轮足底盘 + 双臂**，通过**端到端技能策略（VLA）+ 上层智能体编排（AgentOS）** 完成一系列桌面/作业技能。

目标能力（按优先级）：

| 序 | 任务 | 判定难度 | 动作复杂度 | 定位 |
| --- | --- | --- | --- | --- |
| 1（首发）| 关灯 | 易（灯灭/开关到位）| 单步定点 | 最小闭环验证 |
| 2 | 捡东西丢垃圾桶 | 中（物体是否进桶）| 多步 + 移动 | 闭环成型后 |
| 3 | 把地上东西放桌上 | 中 | 多步 + 移动 | |
| 4 | 叠衣服 | 难（需语义判定 → 云端 VLM）| 高 | 远期，验证云端判定 |

**核心原则：本地优先（Local-First）。** 因目标场景含网络不稳定/断网环境（农业等），凡本地能完成的决策与执行**必须用本地模型**，云端不再是默认路径，而是被本地决策层**按需调用**的可选通道。断网时系统仍能完成"是否动臂"的决策，不整体瘫痪。

**我的职责边界**：本地 LLM（决策脑）+ 整个系统架构（AgentOS 编排层 `agent-llm-vla/src/agent_api/`）+ ASR 语音接入 + 视觉判定（成败裁决）+ **行进路线规划**（决定走哪条路线/途经点，下发路径命令给底盘同学）。**不训练 VLA、不写机械臂技能策略、不写底盘低层运动控制（前进后退转弯的电机驱动）、不实现真实 LLM provider、不碰硬件驱动。**

**两个隔离的下游黑盒**（我只下发命令走接口，不管其内部实现）：
- **机械臂（VLA）** = 同学 A。我下发 `skill_id`，他执行抓取/操作。我完全不管"手臂怎么动"。
- **底盘移动** = 同学 B。我下发**路径/路线命令**，他执行具体前进后退转弯（低层电机控制）。**路线的高层规划归我，轮子的低层驱动归他。**

## 二、团队分工与协作边界

| 角色 | 负责 | 与我的接口 |
| --- | --- | --- |
| **本人** | 本地 LLM 决策脑 + 系统架构（AgentOS 编排：意图识别、指令下发、反馈闭环、失败重试、多轮、安全闸门）+ ASR 接入 + 视觉判定 + **行进路线规划** | 产出 `(intent, ToolCall)`；消费 `VLAExecuteResponse`；向底盘下发路径命令 |
| **VLA / 机械臂 同学（A）** | VLA 服务（技能策略、LeRobot rollout、真机/仿真）—— **黑盒** | 我只调其 HTTP：`POST /v1/execute`，收 `ok/return_code` |
| **底盘 / 移动 同学（B）** | 底盘低层运动控制（把路径命令变成前进后退转弯的电机驱动）—— **黑盒** | 我只下发**路径/路线命令**（颗粒度待与其对齐），收移动执行结果 |
| **LLM 同学**（若有）| 真实 LLM provider | 我只调其 HTTP：`POST /v1/generate` |
| **带教/团队** | 硬件供给、真机策略、摄像头画面来源、云端预算 | 见"待确认项" |

**协作红线**（详见约束手册第五/九节）：
- VLA 与底盘都是黑盒：我只发请求、等结果、查成败、（若支持）取消；**不监控其内部状态、不干预其执行序列**。
- **两条下游边界的分工**：机械臂——我只发 `skill_id`，不管手臂怎么动；底盘——我发**高层路径**（走哪条线/途经点），不管轮子怎么驱动。
- 向下只传明确指令（VLA: `skill_id` + `user_text`；底盘: 路径命令）；**置信度/理由绝不跨 Agent→下游边界下发**。
- 向上只回报执行结果（`ok/return_code`）。
- 换 VLA/底盘/LLM/推理板 = 只改 provider 与配置，**不改 Agent 主流程**。

## 三、系统架构（目标形态）

### 3.1 六层职责链（自上而下 · 判断在 OS 层，执行在底层）

```
[用户语音]
   │
   ▼
① ASR 服务（SenseVoice-Small，本地）        —— 语音→文本。仅转写，不判断。
   │ 文本指令
   ▼
② 本地 LLM 决策服务（本地小模型，OpenAI 兼容 HTTP）
   │    —— 决策脑：意图分类 + 实体抽取，产出 (intent, ToolCall) + 置信度/理由/是否追问
   │ (intent, ToolCall)              ← 黄金契约边界
   ▼
③ AgentOS 层（agent_api/：router + orchestrator + route_planner + main）    ← 我的主战场
   │    · router      ：调②拿决策（或关键词快速路径 / 离线兜底）
   │    · orchestrator：安全闸门(读置信度) → Dry-run → 执行 → 视觉判定 → 重试状态机
   │    · route_planner：复合任务中规划行进路线，产出路径命令（我新增的自有模块）
   │    · main        ：HTTP 入口，Lifespan 健康检查，组装 AgentResponse
   │    ├─ 需动臂 → 下发「明确指令」(skill_id + user_text)  ─────────┐
   │    └─ 需移动 → 路线规划 → 下发「路径命令」(颗粒度待定)  ──┐      │
   ▼                                                          ▼      ▼
④a 底盘/移动服务（另一同学 B · 黑盒）        ④b VLA 服务（vla_service/，:8011，同学 A · 黑盒）
   │    收路径命令，跑低层运动控制             │    收 skill_id，跑策略，返回 ok/return_code
   │    ↓ 电机驱动                            │    ↓ 调 LeRobot rollout
   ▼                                          ▼
⑤a 底盘硬件（轮/足）—— 前进后退转弯，纯执行   ⑤b 策略执行（LeRobot/SmolVLA）→ 机械臂动作序列
                                              │    ↓ 串口
                                              ▼
                                          ⑥ 机械臂硬件（/dev/ttyACM0）—— 纯执行，无判断
```

> **两条下游执行链（并列的黑盒）**：机械臂链（④b→⑤b→⑥，同学 A）与底盘移动链（④a→⑤a，同学 B）。orchestrator 在**复合任务**中按状态机交替调度二者（如"把地上东西放桌上"= 移动→抓取→移动→放下）。**路线规划的决策在我这（route_planner 出路径），底盘同学只做低层执行**。

横向两条（不在底层执行链上，均由 ③ 的 orchestrator 调用）：
- **云端 LLM 服务**（:8012）：当 `intent = llm.general_qa` 时调用（助理问答，如天气/穿搭），本地不推理。
- **视觉判定 Judge**（本地 YOLO）：动作结束后判定任务成败，产出 `{success, detail, evidence}`，构成反馈闭环。

### 3.2 现有代码骨架（实读确认 · 不推翻）

三服务解耦（HTTP）：**Agent API `:8010` / VLA Service `:8011` / LLM Service `:8012`**。

```
用户输入 → Agent API(:8010) → route_user_text()
                               ├─ 命中 → VLA Service(:8011) → mock/真实技能
                               └─ 未命中 → LLM Service(:8012) → mock/真实大模型
```

现状（阶段0/1 已验收）：环境 + mock 联调 100% 通；源码实现 0%（`router.py` 仍 28 行纯关键词，`orchestrator.py` 未创建）。

### 3.3 设计原则（不可推翻）

- **认知-行动解耦**：LLM 只做"做什么"（intent + 参数）；状态机做"怎么做"（执行序列）。**永远不让 LLM 直接控制物理执行序列。**
- **Provider 可插拔**：ASR / 本地 LLM / 云端 LLM / VLA / 视觉判定 都是可替换 provider，Agent 只依赖契约不依赖实现。
- **契约稳定**：`route_user_text() → (intent, ToolCall)` 是黄金签名，任何升级都不破坏它；`schemas.py` 只向后兼容新增。
- **硬边界状态机 > 自由 ReAct**（OK-Robot 实证：171 次真实家庭实验，简单线性状态机成功率 58.5%（零样本），干净环境 82.4%，远超 LLM 自由推理；StateAct 比 ReAct 效率高 40%——31.49 步 vs 19.11 步）。
- **安全前置**：置信度闸门（<0.75 拒绝下发，走 CONFIRMING 反问）+ 危险动作分级（low/medium/high/critical，high 以上二次确认）+ dry-run 物理预演（碰撞/关节极限检查）+ 启发式恢复（查表不让 LLM 重新规划）。

## 四、技术栈与选型（已定方向）

| 层 | 选型 | 依据 |
| --- | --- | --- |
| **ASR（语音→文本）** | SenseVoice-Small | 254MB、纯 CPU 实时、中文 CER~8%、20× 实时；短指令最优。备选 Whisper.cpp |

| ┗ K3（8GB）主力层 | qwen2.5:1.5B（Q4 ~1.1GB，Tool 0.800，JSON 合规 95.7%）| 最稳选择：中文生态最成熟、JSON 合规率最高；追 benchmark 可上 qwen3:1.7B（0.960，但思考模式延迟 ~10s）|
| ┗ 笔记本（16GB）开发 | 可跑 7B 级验证（qwen2.5-7B Q4 ~4.5GB）| 开发环境模拟边缘板 |

| **配置** | YAML 层级覆盖（base + env-specific，`CHAOFENG_ENV` 选择）+ Pydantic 校验 | 笔记本/310B/K3 无缝切换 |

## 六、代码规则（Python / FastAPI）

- **契约优先**：改动前先确认不破坏 `(intent, ToolCall)` 签名与 `schemas.py` 既有字段。
- **风格**：`from __future__ import annotations`；全量类型标注；Pydantic V2 语法（`model_dump()` / `ConfigDict`）；新增 schema 字段必须带默认值。
- **KISS / DRY**：能复用现有 `VLAClient` / `LLMClient` / `get_section` / 现有字段就复用，**零容忍 copy-paste**。
- **保护调用链**：改函数签名同步更新所有调用点（如 `main.py:39` 解包 `route_user_text`、`main.py:46` 读 `arguments["skill_id"]`）。
- **不硬编码**：`max_retries`、置信度阈值、模型路径、端口、开关一律走 `app.yaml` / 环境变量。
- **错误处理**：关键路径（HTTP 调用、JSON 解析、LLM 输出）必须 try-catch + 降级 + 日志。
- **安全**：不硬编码密钥/checkpoint；网络暴露端点标注鉴权缺失；真机动作前 dry-run；危险动作二次确认。
- **提交纪律**：commit/PR **只署名本人，禁止任何 AI 共同署名或生成标记**。
- **收尾清理**：临时文件、废弃代码、未用导入、调试日志。

## 九、术语表

| 术语 | 含义 |
| --- | --- |
| AgentOS | 本项目编排层（agent_api），负责听指令→选技能→下发→盯执行→收反馈→重试 |
| VLA | Vision-Language-Action，端到端技能策略（机械臂执行器），本项目中为黑盒 HTTP 服务 |
| `(intent, ToolCall)` | 黄金契约：router 的输出签名，意图字符串 + 工具调用对象 |
| A3 契约 | `ToolCall.metadata` 新增（默认 None），含 confidence/reason/need_clarify，仅 OS 层消费不下发 VLA |
| Judge | 视觉判定器，动作后裁决任务成败，契约 `{success, detail, evidence}` |
| Dry-run | 预演：真机前的几何碰撞/关节极限检查，通过才下发 |
| DAG 状态机 | 编排核心：IDLE/DISPATCH/CONFIRMING/DRY_RUN/EXECUTE/JUDGE/RETRY/REPORT_FAIL/COMPLETE/CANCELLED；YAML 配置驱动状态转换表 |
| Code as Policies | 复杂任务让 LLM 生成受限 Python 代码（循环/条件）交安全沙箱执行，而非单步 JSON |
| Local-First | 本地优先：本地能做的必须本地做，云端仅按需 |
| GGUF | llama.cpp 的模型权重格式，同格式换板不换接口 |
| MCP | Model Context Protocol，动态能力发现协议，本项目预留迁移路径 |
| 复杂度路由器 | 按关键词判任务复杂度（简单/中等/复杂），决定走 JSON 快速路径还是 Code as Policies |
| 启发式恢复 | 判定失败时查表执行预定义恢复动作（微调角度/后退重试），不让小模型重新规划（调研明确反对 LLM 重规划：小模型面对负反馈易死循环）|
| route_planner | 行进路线规划模块（我新增），在复合任务中决定走哪条路线/途经点，产出路径命令下发底盘同学；路线高层规划归我，轮子低层驱动归底盘 |
| 两个下游黑盒 | 机械臂（VLA，同学 A，收 skill_id）与底盘移动（同学 B，收路径命令），均只调接口不管内部实现 |
| 复合任务 | 需移动 + 动臂交替的任务（如"把地上东西放桌上"），由状态机在 route_planner 与 VLA 间交替调度；执行序列预定义（task_templates.yaml），LLM 只选模板+填参数 |
