## 二、状态机正式定义（`orchestrator.py` 的唯一设计图）

> 消歧：01术语表列全集 10 状态、02/03 列子集。**本节为正式定义**：全集枚举 + 迁移表 + **分阶段落地子集**（S3 只实现哪些、S4 加哪些、S8 加哪些）。实现者照此建 `OrchestratorState` 枚举与迁移函数。

### 2.1 状态全集（枚举 · 与 01术语表对齐）

```python
class OrchestratorState(str, Enum):
    IDLE        = "idle"          # 初始/空闲
    DISPATCH    = "dispatch"      # 拿到 (intent, ToolCall)，准备派发
    CONFIRMING  = "confirming"    # 置信度 < 阈值，反问澄清（S8 起）
    DRY_RUN     = "dry_run"       # 几何/碰撞预演（真机前，S9 起有真实语义）
    EXECUTE     = "execute"       # 下发 VLA / 底盘，等执行结果
    JUDGE       = "judge"         # 视觉判定成败
    RETRY       = "retry"         # 失败未超次，回 EXECUTE
    REPORT_FAIL = "report_fail"   # 失败超次，出失败报告
    COMPLETE    = "complete"      # 成功终态
    CANCELLED   = "cancelled"     # 用户/系统取消终态
```

### 2.2 迁移表（触发条件 → 目标状态）

| From | 触发条件 | To | 阶段 |
|---|---|---|---|
| IDLE | 收到请求 | DISPATCH | S3 |
| DISPATCH | `confidence >= 阈值` 或 无置信度信息 | EXECUTE（经 DRY_RUN） | S3 |
| DISPATCH | `confidence < 阈值`（0.75） | CONFIRMING | **S8** |
| DISPATCH | tool_type=LLM（问答类） | 直接走 LLM 分支→COMPLETE | S3 |
| CONFIRMING | 用户补充/确认 | DISPATCH（重评） | S8 |
| DRY_RUN | 预演通过 | EXECUTE | S9（S3–S4 mock 阶段 dry_run=透传标志，不做真实几何校验） |
| DRY_RUN | 预演不通过 | REPORT_FAIL | S9 |
| EXECUTE | 拿到 `VLAExecuteResponse` | JUDGE | S3 |
| EXECUTE | HTTP 调用异常/超时 | REPORT_FAIL（或降级，S8+） | S3 |
| JUDGE | `执行 ok` 且 `判定 success` | COMPLETE | S3 |
| JUDGE | 任一为假 且 `attempts < max_retries` | RETRY | **S4** |
| JUDGE | 任一为假 且 `attempts >= max_retries` | REPORT_FAIL | **S4** |
| RETRY | 回到执行（attempts+1） | EXECUTE（经 DRY_RUN） | S4 |
| 任意非终态 | 用户取消 / 关键异常 | CANCELLED | S12（取消能力）先留位 |

**成败合议**（JUDGE 的判定逻辑，S3 落地）：`执行层 ok=true` **AND** `视觉判定 success=true` → COMPLETE；任一为 false → 进重试/报告分支。

### 2.3 分阶段落地子集（防过度设计：不一次实现全集）

| 阶段 | 落地状态 | 落地迁移 | 明确不实现 |
|---|---|---|---|
| **S3**（反馈闭环骨架） | DISPATCH, EXECUTE, JUDGE, COMPLETE, REPORT_FAIL | 派发→执行→判定→成/败终态（**无重试**，失败直接 REPORT_FAIL） | RETRY / CONFIRMING / 真实 DRY_RUN |
| **S4**（重试状态机） | +RETRY | JUDGE 失败→RETRY→EXECUTE；超次→REPORT_FAIL；dry_run 贯穿每次 attempt | CONFIRMING / 真实 DRY_RUN |
| **S8**（决策脑+闸门） | +CONFIRMING | DISPATCH 读 confidence <0.75→CONFIRMING | 真实几何 DRY_RUN |
| **S9**（真闭环） | DRY_RUN 转真实语义 | 真机前几何/碰撞预演 | — |
| **SR3**（复合任务） | 复用上述，按段（NAV/VLA）循环 | 移动段与动臂段各跑一遍"执行→判定→重试" | 真实路径规划算法 |

> 🔩 **dry_run 贯穿**：S3–S4 mock 阶段，`dry_run` 是一路透传到 `VLAClient.execute_skill(dry_run=...)` 的**布尔标志**（现有 `main.py` 已透传），不做真实几何校验；DRY_RUN 状态的"真实预演语义"到 S9 才填。测试断言：mock 阶段每次 attempt 的下发请求 `dry_run=true`（05 S4 已列）。

## 三、接口契约集约（一处给全签名 · 实现直接照抄骨架）

> 以下为**目标签名**，非现状。用现有代码风格（`from __future__ import annotations`、`str | None`、V2 Pydantic、中文 docstring，见 04 第三节）。

### 3.1 orchestrator 编排入口（S3 新建）

```python
def orchestrate(
    intent: str,
    tool_call: ToolCall,
    *,
    dry_run: bool = False,
    user_text: str,
    metadata: dict[str, Any] | None = None,
    judge: "Judge | None" = None,          # 缺省注入 MockJudge（provider 可插拔）
) -> dict[str, Any]:
    """编排入口：拿 (intent, ToolCall) → 执行 → 判定 → (S4起)重试，返回可塞进
    AgentResponse.raw_result 的编排结果（含 final_state / attempts / judge 详情）。"""
```

- `main.py` 的 `/v1/chat` 改为：`route_user_text` → `orchestrate(...)` → 组装 `AgentResponse`。🔩 **VLA/LLM 分支逻辑整体迁移进 orchestrate，不新旧并存**（02 S3.1）。
- 返回值先落 `raw_result`（不动 schema）；确需强类型再升 `AgentResponse`（需二次确认，见 SSoT 1.3 变更规则）。
- 🔩 **A3 私有字段不下发**：`confidence/reason/need_clarify`（S8 起在 `ToolCall.metadata`）仅 orchestrate 内消费，下发 VLA 的请求体绝不含它们。

### 3.2 Judge 视觉判定协议（S3 定义 + MockJudge，S6 加 YoloJudge）

```python
class Judge(Protocol):
    """视觉判定协议：动作后裁决任务成败。provider 可插拔。"""
    def judge(self, intent: str, tool_call: ToolCall,
              exec_result: VLAExecuteResponse,
              metadata: dict[str, Any]) -> dict[str, Any]:
        """返回 {"success": bool, "detail": str, "evidence": dict}。"""

class MockJudge:
    """mock 判定器：期望值由 metadata['mock_judge_success'] 注入，默认 True；
    evidence 恒 {}。用于在 mock 阶段主动触发失败/成功两条分支。"""
```

- 🔩 期望值**走 `UserRequest.metadata` 注入**（已确认 schema 支持），**不硬编码**——这是 S4 失败注入测试的设计基础。
- S6 的 `YoloJudge` 实现同协议，orchestrator 按 `agent.judge.provider` 配置切换，**编排代码零改**。

### 3.3 RoutePlanner 协议 + NavClient（SR，第一月只留占位）

```python
class RoutePlanner(Protocol):
    """行进路线规划协议（我的自有模块）：复合任务中出高层路径命令。"""
    def plan(self, goal: dict[str, Any],
             context: dict[str, Any]) -> "NavCommand": ...

class MockRoutePlanner:
    """直通桩：输入目标 → 输出占位路径命令（mock 恒可达）。不实现真实几何/图搜索。"""
```

- `NavClient`（`tool_clients.py` 新增，**与 `VLAClient` 平级、独立下游**，🔩 不复用 VLAClient）：`send_nav_command(cmd: NavCommand) -> NavResult`，mock 阶段恒 `ok=true`，复用 `common.http_client`。

- 🔩 route_planner 只出高层路径，**绝不写轮子低层驱动**（电机控制归 B）。

## 四、风险驱动的实现顺序（依赖硬约束不破 + 高风险先加护栏）

> 依赖硬约束（02 已定，不可违反）：S3 依赖 S2 的 `(intent, ToolCall)`；S4 依赖 S3；S8 复用 S2 插槽；S9 依赖 S7+S8。
> 本节在**不破依赖**前提下，按"易碎/不确定/被卡"给每步配护栏与顺序，而非无脑平铺。

### 4.1 风险画像

| 阶段 | 风险类型 | 为什么 | 护栏（先做） |
|---|---|---|---|
| **S3** | 🔴 **最易碎**（回归风险） | 要把唯一在跑的 `main.py` VLA/LLM 分支**整体迁移**进 orchestrate，稍错即破现有闭环 | 迁移前先存**基线快照**（改造前 smoke 四项逐字段输出存档），迁移后逐字段对比等价（05 S3 迁移等价）|
| **S8** | 🟡 **最不确定**（外部模型） | JSON 合规、fallback、置信度都依赖真实 LLM 行为，mock 测不出真值 | Fallback 链 + 契约不泄漏断言先写；置信度阈值用测试集校准；断网降级到关键词是命脉 |
| **SR3** | ⚫ **最被卡**（外部依赖） | 命令颗粒度待同学 B 对齐，未定就落地会返工 | SR1/SR2 只做接口占位+直通桩；SR3 标 `[!]` 阻塞，对齐前不落地 |
| **S2** | 🟢 低风险（独立、可回归） | 纯增强 router，不动签名，smoke 可即时回归 | 变体集量化命中率，别用单句冒充 |

### 4.2 推荐实现顺序（第一月）

```
① S2 意图识别增强   ← 最低风险、独立、立即可回归，先拿它建"变体集+回归"的测试习惯
② S3 反馈闭环骨架   ← 最易碎，投入最大验证（基线快照+迁移等价）。先只到 COMPLETE/REPORT_FAIL，不做重试
③ S4 失败重试状态机 ← 在 S3 稳态上加 RETRY 一条边 + dry-run 贯穿；靠 MockJudge 失败注入测全分支
④ S5 ASR / S6 YOLO  ← 引入真实依赖，与 S2/S3 联合测（ASR 误差×意图、假阳/假阴）
并行：SR1 底盘接口占位（无依赖，任意时点可做，开关默认关，不影响主线回归）
```

> 关键判断：**S3 排在 S2 之后、S4 之前**符合依赖；但把"最大验证投资"押在 S3（而非按行数看似简单就轻放）是风险驱动的核心——迁移等价性一旦破，后面全建在流沙上。

### 4.3 每步的"先加护栏"清单（写码前先备）
