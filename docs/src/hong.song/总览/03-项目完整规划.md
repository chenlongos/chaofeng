## 一、总体架构决策（贯穿所有阶段）

### 1.1 目标形态（local-first）

```
[音频] → ASR(SenseVoice,本地) → 本地LLM决策脑(小模型)
                                     ├─ 动臂: (intent,ToolCall) → VLA服务(HTTP) → 策略 → 机械臂   [下游黑盒A]
                                     ├─ 移动: route_planner(我) → 路径命令 → 底盘服务(HTTP) → 电机  [下游黑盒B]
                                     └─ 问答: intent=llm.general_qa → 云端LLM(按需)
                          orchestrator: 安全闸门 → DRY_RUN → EXECUTE → JUDGE(本地YOLO) → RETRY/REPORT_FAIL
                          复合任务: 状态机在 route_planner(移动) 与 VLA(动臂) 间交替调度
```
> **两个隔离的下游黑盒**：机械臂（VLA，同学A，收 `skill_id`）与底盘移动（同学B，收路径命令）；`route_planner` 是我的自有模块——路线高层规划归我，轮子低层驱动归 B。详见 SR 节。

### 1.2 五条不可动摇的架构原则 🔩

1. **契约黄金律**：`route_user_text()` 恒输出 `(intent: str, ToolCall)`，下游 VLA/LLM 零改动。
2. **判断在 OS 层，执行在底层**：意图+置信度 100% 在 AgentOS（本地LLM+orchestrator）；向下只传 `skill_id`(+user_text)，置信度/理由**绝不跨 ③→④ 边界**下发 VLA。
3. **provider 可插拔**：ASR/LLM/VLA/Judge 都是可替换 provider，换后端只改 provider+配置。
4. **本地优先**：本地能做的必须本地做，云端仅被本地决策层按需调用；断网不瘫痪。
5. **LLM 做"做什么"，状态机做"怎么做"**：绝不让 LLM 控制物理执行序列。

### 1.3 为什么是这套架构（调研支撑）

- **多服务解耦**：VLA 控制循环 10–50Hz vs LLM 推理 1–3s，物理解耦防推理阻塞伺服（调研报告 SOA 论证）。
- **硬边界状态机 > ReAct**：OK-Robot 171 次真实实验证明简单线性状态机成功率远超 LLM 自由推理；纯 ReAct 易因传感器噪声陷入"推理-重试"死循环。
- **小模型够用**：Qwen 家族 1.5B–3B 在 Tool Calling / 意图分类 / JSON 合规上达标（qwen2.5-1.5B JSON 合规 95.7%）。

## 二、第一月（mock）逐阶段规划

### S2 · 意图识别增强

**目标**：换说法不漏判；守签名；为本地 LLM 决策脑铺路（S8 直接复用此插槽）。

**具体做法**：
1. 文本归一化（全角半角/大小写/标点）——只作用于匹配，`user_text` 原文照传下游。
2. 扩 `app.yaml` `agent.intent_rules` 同义关键词（配置驱动，改允许改的 `agent.*`）。
3. LLM 分类兜底插槽：默认 `enabled=false`；开时关键词未命中走 LLM 分类；🔩 产出 `arguments` 键集必须含 `skill_id`。

**可选方式与利弊**：

| 路由策略 | 做法 | 利 | 弊 | 取舍 |
|---|---|---|---|---|
| **A 关键词主 + LLM 插槽（本期）** | 关键词优先，未命中才 LLM | 稳、快、mock 即验收、零依赖 | 换说法仍靠扩词表 | ✅ 第一月选此 |
| B 本地 LLM 主 + 关键词兜底 | LLM 先决策，关键词做快速路径/离线兜底 | 泛化强，是目标形态 | 依赖本地 LLM 就绪（S8） | ✅ 第二月切此（讨论已定选 A 路由策略） |
| C 关键词主 + LLM 仅消歧 | 只在歧义时调 LLM | 调用少 | 覆盖不全 | 落选 |

**待进一步讨论**：
- ✅ **已定**：首发两任务（关灯+捡球）每个扩 8-12 个高频关键词（动词变体、口语化、否定表达）；其余任务只放 2-3 个核心词，交给 S8 的 LLM 处理边缘情况。后续增加任务再细化。

### S3 · 反馈闭环骨架

**目标**：从"做完即忘"升级为"收状态+判成败"；抽 orchestrator；Judge 协议可插拔。

**具体做法**：
1. 新建 `orchestrator.py`，`main.py` 只接线（VLA/LLM 分支逻辑整体迁移，🔩 不新旧并存）。
2. 复用 `VLAExecuteResponse.ok/return_code` 判执行层成败。
3. `Judge` 协议产 `{success, detail, evidence}`；先实现 `MockJudge`（期望值走 `metadata` 注入）。

**可选方式与利弊**：

| 判定信息落点 | 利 | 弊 | 取舍 |
|---|---|---|---|
| **先落 `raw_result`（本期）** | 不动 schema、零风险 | 结构不强类型 | ✅ 默认走这个 |
| 升 `AgentResponse` 加 `success/attempts` 字段 | 强类型、前端好用 | 动 schema（需二次确认） | ⚠ 确需时才动，只加带默认值字段 |

**待进一步讨论**：
- ✅ **已定**：S3 用 `dict` 泛型（`evidence: dict[str, Any] = Field(default_factory=dict)`），S6 接 YOLO 时定义 evidence 内部的"推荐格式"（文档约定，不强制 schema）。后续如需强类型再升级（向后兼容，只加不改）。不过早锁死，避免 S6 时改 schema 违反红线。

### S4 · 失败重试状态机

**目标**：失败能重试、超次能报告，全程 dry-run 安全。

**具体做法**：DISPATCH→EXECUTE→JUDGE→分支；`max_retries` 走配置；attempt 详情落 `raw_result`。

**可选方式与利弊**：

| 重试恢复策略 | 做法 | 利 | 弊 | 取舍 |
|---|---|---|---|---|
| **原样重试（本期 mock）** | 失败就同参数再来一次 | 简单，mock 够演 | 真机可能同样失败 | ✅ 第一月 |
| 启发式恢复（进阶） | 查表调整（接近角度/后退2cm） | 真机有效，抑制小模型重复输出陷阱 | 需真机+查表设计 | S11，🔩 不让 LLM 重新规划 |
| LLM 重新规划 | 失败喂回 LLM 再想 | 灵活 | ⚠ 小模型面对负反馈易死循环 | ❌ 调研明确反对 |

**待进一步讨论**：
- ✅ **已定**：`max_retries = 3`，所有环境统一，不搞差异化配置（mock 阶段 VLA 假执行区别不大；真机阶段第一次失败可能是偶然因素，3 次更稳）。

### S8 · 本地 LLM 决策脑接入 ⭐ 第二月核心

**目标**：S2 的"LLM 分类插槽"从默认关升为主决策层，实现 A3 契约 + 安全闸门。

**具体做法**：
1. 笔记本起 `llama-server`（qwen2.5-1.5B GGUF），OpenAI 兼容 HTTP。

3. A3 契约：`ToolCall.metadata` 新增（默认 None），含 `confidence/reason/need_clarify`。
4. prompt 存 YAML（极简 system ~20t + tools + rules + few-shot 3 例覆盖 正常/问答/追问）。
5. JSON 解析 + Fallback 链（非纯 JSON/截断/字段缺失 → try-catch → 关键词兜底 → 记录 fallback 事件）。
6. orchestrator 安全闸门读 `confidence` < 阈值 → CONFIRMING（反问），不下发。

**可选方式与利弊**：

| 决策模型档位 | 平台 | Tool 得分 | 取舍 |

| qwen2.5-1.5B | K3/笔记本 | 0.800，JSON 95.7% | ✅ 最稳起点 |
| qwen3-1.7B | K3 | 0.960（冠军） | ⚠ 思考模式延迟 ~10s |
| qwen3-0.6B | 310B(4GB) | 0.880 | ✅ 310B 上限 |

### SR · 行进路线规划（route_planner）｜复合任务前置，颗粒度待定

> **定位**：这是「我」的自有模块，不是下游黑盒。夹在 orchestrator 与底盘同学（B）之间——**路线高层规划归我，轮子低层驱动归 B**。只在**复合任务**（需移动+动臂交替，如"把地上东西放桌上"）里被状态机调用；单步定点任务（首发关灯）不涉及移动，故 SR 在首发闭环里是**空转/直通**。

**目标**：在复合任务的移动子步骤里，由 `route_planner` 决定走哪条路线/途经点，产出**路径命令**下发底盘同学；收其移动执行结果，交回状态机继续下一步（动臂）。

**与其他阶段的关系（🔩 复合任务中交替，非平级独立下游）**：
- 状态机在一个复合任务里按序调度：`route_planner 出路径 → 底盘执行移动 → VLA 动臂 → …`，二者在**同一任务的不同状态**里交替出现。
- 首发任务（关灯）单步定点，**不触发 SR**；SR 的真实价值从第二个任务（捡东西丢垃圾桶，需移动到物体+移动到桶）起显现。
- 复用现有编排骨架：SR 是 orchestrator 状态机新增的一类"移动子步骤"，不新起服务、不破 `(intent, ToolCall)` 签名。

**具体做法（mock 阶段只搭骨架，不实现真实规划算法）**：
1. 定义 `NavClient` 接口占位（与 `VLAClient` 平级的下游 client），封装"下发路径命令 → 收移动结果"的 HTTP 调用；mock 阶段返回恒成功。
2. 新增 `nav` 类意图 / ToolCall 分支（如 `robot.move_to`），🔩 `arguments` 键集与 VLA 路径同构原则一致（下游读什么键要与 B 对齐后写死）。
3. `route_planner` 模块先做**直通桩**：输入目标位置 → 输出占位路径命令（mock 恒可达），把接口拉通、状态机能交替调度即可。
4. 真实规划算法（几何路径/避障/途经点）**留到颗粒度与 B 对齐后再实现**，本期不写。

**可选方式与利弊（下发底盘的命令颗粒度 —— ⚠ 核心待定，决定 route_planner 工作量）**：

| 命令颗粒度 | 做法 | 我这边工作量 | 利 | 弊 | 取舍 |
|---|---|---|---|---|---|
| **A 语义目标**（"去客厅"/"到桌前"） | 我只发语义目标，B 内部自己规划路径 | 最小（几乎不做路径规划） | 我轻、接口简单、耦合低 | 路径决策权在 B，我"规划路线"的职责被架空 | ⚠ 待与 B 对齐 |
| **B 路径点序列 waypoints**（坐标点列） | 我做几何路径规划出途经点，B 只跟点走 | 大（要做路径规划+地图+坐标系） | 路线决策权在我，符合职责定位 | 依赖地图/定位/坐标系来源，工作量大 | ⚠ 待与 B 对齐 |
| **C 离散移动指令**（前进 X 米/左转 Y 度） | 我拆成离散动作序列，B 逐条执行电机 | 中 | 不需要全局地图 | 累积误差大、无闭环校正 | ⚠ 备选 |

> 三选一取决于**地图/定位从哪来、B 的接口能力**。在与 B 对齐前，代码里只留 `NavClient` 接口占位 + `nav` 意图占位，**不预设颗粒度**，避免过早锁死返工。

**待与底盘同学 B 对齐（🔩 全部需与 B + 带教确认）**：
- ⚠ **命令颗粒度选 A / B / C？** —— 直接决定 route_planner 是"薄转发"还是"真做路径规划"。
- ⚠ 收哪些移动执行结果字段？（到位与否 / 实际位姿 / 失败原因 → 决定 JUDGE 和 RETRY 怎么接移动失败）
- ⚠ 坐标系 / 地图从哪来？（B 提供 / 共享 SLAM / 预设地图 → 决定 waypoints 方案是否可行）
- ⚠ 移动失败的重试策略？（复用 S4 状态机的 RETRY，还是移动有独立恢复逻辑）
- ✅ **已定**：复合任务的任务分解用预定义任务模板（`task_templates.yaml`），LLM 只决定"用哪个模板"和"填什么参数"，Phase 3 复杂任务用 Code as Policies。结合 OK-Robot 实证（硬边界状态机 > LLM 自由推理），不让 LLM 自由拆分执行序列。

## 四、进阶阶段规划（有余力/后续）

### S10 · 边缘部署（310B → K3）

**具体做法**：310B 起 LLM 服务 + OpenAI 兼容端点 → YAML 环境切换 → 混沌测试 → HIL 仿真 → Docker Compose → 迁 K3。
