# 新同学任务规划

> 欢迎加入嘲风机器人项目。这份文档说明你要负责的两块工作、每块的目标和验收标准，以及和现有系统的边界。不含时间表——先把"做什么、做到什么程度"看清楚，节奏我们再一起对齐。

---

## 〇、一分钟看懂你在整个系统里的位置

我们的机器人软件分好几层。从"人说话"到"机械臂动"，大致是这样一条链：

```
用户语音/文本
  → 语音识别(ASR)
  → 意图理解 + 编排大脑(状态机：决定调哪个能力、要不要重试、成没成)
  → 执行(机械臂动作 / 问答)
  → 判定(动作做完后，看看到底成没成)
  → 反馈给用户
```

这条链的**"编排大脑"已经由我（负责人）实现并跑通了**——它决定"下一步调哪个能力、动作失败要不要重试、整个任务算成功还是失败"。这部分你不需要动，也请不要动（后面会讲为什么，这是保证系统不乱的关键）。

**你要做的是两块"大脑还没有、但系统很需要"的新能力**，它们都是独立的、边界清晰的模块：

1. **World Model（世界状态层）**——让机器人"记住现在世界长什么样"
2. **视觉事实服务（判定内核）**——让机器人"看一眼，回答某件事成没成"

这两块都是**真正从零开始的新东西**，做出来就是你的成果，而且和我的编排大脑之间有明确的接口，你只管把接口后面的东西做好，不用担心牵一发动全身。

## 一、任务包一：World Model（世界状态层）

### 这是什么

一句话：**一个"结构化、可查询、带时间和置信度"的世界状态记录。**

它不是相机画面，也不是数据库。它回答的是这一类问题：

- 现在场上有哪些物体？（球、篮子、灯……）
- 它们在哪？（坐标、在哪个坐标系下）
- 它们什么状态？（球在桌上 / 被夹住 / 已放进篮子 / 丢了）
- 这个信息有多可信？（置信度）
- 最后一次看到它是什么时候？（last_seen，太久没看到的信息要打折）

举个直观例子。相机看到一个球，World Model 里应该是这样一条记录：

```json
{
  "id": "tennis_ball_01",
  "class": "tennis_ball",
  "pose": { "frame": "base_link", "x": 0.48, "y": 0.25, "z": 0.03 },
  "state": "on_table",
  "confidence": 0.91,
  "last_seen": "2026-07-15T12:55:32Z"
}
```

关键点：如果机械臂动作时挡住了球，相机看不到了，World Model **不能马上把球删掉**，而应该把它标成"被遮挡、置信度下降"，并记住"上次是什么时候看到的"。这就是"模型"的含义——它是系统对现实的**当前判断**，不是现实本身。

### 具体要做什么

1. **定义状态结构**：物体（id / 类别 / 位姿 / 状态 / 置信度 / last_seen）、机器人自身状态（后续双臂时扩展成左臂/右臂/共享工作区）。
2. **提供查询接口**：`get_scene()`（看全场）、`get_object(id)`（查单个物体）等只读查询。
3. **更新逻辑**：拿到新的感知结果时，怎么更新已有物体、怎么新增、怎么处理"同一个物体连续两帧要认成同一个"（id 稳定性）。
4. **置信度衰减**：信息会过期。静止的篮子慢慢衰减，滚动的球要快速衰减。太久没更新的物体，置信度降下来。
5. **状态机**：物体状态怎么流转，比如 `on_table → 被抓取中 → 被夹住`（需要"夹爪反馈 + 视觉确认"才能确认抓住，不能因为"发了抓取命令"就断言抓住了）。

### 边界（重要，别越界）

- **World Model 只管"世界现在什么样"**，它不决定"下一步做什么"——那是编排大脑的事。
- **它和"记忆"不是一回事**。有个容易混的概念叫 Memory（记用户说过什么、用户偏好、历史任务），那块是我负责的，属于编排层。你的 World Model 只管**物理世界的当前状态**（物体在哪、机器人什么姿态），不碰对话历史和用户偏好。
- 高频的关节角度、电机状态那种硬件底层数据，**不要一股脑塞进来**——World Model 只存"任务相关的、压缩过的"状态，不是硬件监控面板。

### 验收标准

- 能对至少 4 个实体（左臂、右臂、球、篮子）建模、查询、更新。
- 抓取后状态正确流转（抓取中 → 抓住 / 失败回滚）。
- 置信度会随时间衰减，遮挡时物体不会被直接删除而是降置信度。
- 有单元测试覆盖：新增物体、更新物体、置信度衰减、遮挡处理。

## 二、任务包二：视觉事实服务（判定内核）

### 这是什么

一句话：**机械臂做完一个动作后，"看一眼，回答这件事的客观结果"。**

比如机器人执行了"把球放进篮子"，动作做完后，这个服务负责回答："球现在在不在篮子里？"

**好消息：这个服务的外壳、接口、调用方式我已经全部搭好并冻结了。** 系统里已经有：

- 一个独立的判定服务（`judge_service`），带健康检查、HTTP 接口
- 冻结的请求/响应格式（`JudgeRequest` / `JudgeResponse`）
- 一个 mock 版本（假的，恒定返回成功/失败，用来联调）
- 一个**真实内核的占位**（`RewardClassifierProvider`）——**这就是你要填的地方**

**你要做的就是把这个占位填成真的**——把 mock 换成真正能"看图判断"的内核，而接口一个字都不用改。我这边调用你的代码是走 HTTP 的，你把内核做好，我这边零改动。

### 具体要做什么

判定的响应格式是固定的三个字段（已冻结，不要改）：

```json
{
  "success": true,           // 这件事成没成（客观事实）
  "detail": "球已在篮子内",   // 人能看懂的说明
  "evidence": {}             // 判定依据（检测框、置信度、用的哪一帧等）
}
```

你要实现的"真实内核"，可以从下面选路线（我们再一起定，主线建议 RewardClassifier，因为它和我们的机械臂策略是同一套生态）：

- **RewardClassifier**（主线推荐）：LeRobot 自带的奖励分类器，判断"当前这一帧算不算任务成功"。和我们的 SmolVLA 动作策略同生态，天然衔接。
- **视觉检测（YOLO 等）**：检测"物体在不在目标区域"，比如球的框和篮子的框有没有重叠。位置类任务很直观。
- （备选）多模态大模型看图问答：快，但慢/贵/不稳，只适合早期原型。

`evidence` 字段现在留空，等你做真实内核时，把判定用到的检测框、置信度、帧引用填进去，方便排障。

### 边界（这条是红线，务必看清）

**判定服务只回答"世界是什么状态"，不决定"那怎么办"。**

说清楚点：

- ✅ 你负责回答：**球在不在篮子里 / 夹爪夹住没 / 灯灭没 / 置信度多少**——这些是**客观事实**。
- ❌ 你**不负责**决定：**这次任务算不算成功 / 要不要重试 / 下一步该干什么**——这些是**编排大脑的事**（我负责）。

为什么要卡这条线？因为一旦判定服务开始"替系统做决定"，它就侵入了编排大脑的逻辑，两边职责一乱，系统就没法维护了。你只管**报事实**，"根据事实怎么办"交给上层。你返回的 `success` 只是"事实层面的成功信号"，最终任务成败由编排层综合执行结果 + 你的判定一起裁定。

### 验收标准

- 真实内核能对首发场景（捡球、关灯）做出判定，替换 mock 后系统闭环仍跑通。
- 判定准确率达到约定阈值（具体数值我们一起定）。
- 响应延迟可接受（P95 有个上限，我们一起定）。
- `evidence` 里带上判定依据，排障时能看到"为什么判成功/失败"。
- 接口格式和冻结的 `JudgeResponse` 完全一致（不改契约）。

## 三、两块工作和整个系统怎么拼起来

用一句话划清三方职责，记住这句就不会越界：

> **视觉事实服务答"世界现在什么状态"，World Model 答"世界的状态记录/历史"，编排大脑答"下一步调哪个能力、成没成、要不要重试"。**

三块各管一段，通过**冻结的接口**通信，谁都不碰谁的内部实现。你把接口后面的东西做好就行，接口本身不用动，也请不要动。

有任何"这个到底归谁做"的疑问，先问，不要自己跨界改动——尤其是编排大脑（状态机、意图路由、重试逻辑）那部分，那是系统的核心，改动前一定先对齐。

## 四、上手建议

1. **先读代码，别急着写**：把 `judge_service`（你的任务包二的外壳）和现有的 mock 判定跑起来，理解请求怎么进、响应怎么出。
2. **先让 mock 跑通闭环**：确认你能在本地把整条链跑起来（哪怕全是假数据），再开始填真东西。
3. **契约不动，只换内核**：你的所有工作都是"在冻结的接口后面填真实现"，接口格式如果你觉得需要改，先找我讨论，不要直接改。
4. **每块都配测试**：World Model 和判定内核都要有单元测试，这是验收的一部分，也是保证你的东西不被别人误改的护栏。
